응급, 의료, 날씨, 안전 질문은 UMMAYA의 첫 사용 사례로 적합합니다. 대부분 공개 정보에서 시작하기 때문입니다. 사용자는 어느 기관이나 portal이 데이터를 갖고 있는지 먼저 알지 않아도 가까운 병원, 공공 경보, 날씨, 도로 위험, 안전 안내를 질문할 수 있어야 합니다.

동시에 경계도 분명해야 합니다. UMMAYA는 공개 정보를 찾고 요약할 수 있지만, live official path가 증명되지 않는 한 진단, triage, 응급 출동, 병상 보장, 개인 의료기록 접근을 주장하면 안 됩니다.

## 좋은 프롬프트

좋은 안전 프롬프트는 개인 정보를 요구하지 않으면서 `locate`와 `find`를 선택할 수 있는 context를 줍니다. 장소, 상황, 필요한 공개 정보의 종류가 들어가야 합니다.

```text
동아대 승학캠퍼스 근처에서 지금 갈 수 있는 응급실이나 야간 진료 정보를 공식 정보 기준으로 찾아줘.
```

```text
부산 사하구 오늘 호우나 도로 위험 정보가 있는지 공공 데이터 기준으로 확인해줘.
```

이런 프롬프트는 공개 lookup을 요청하기 때문에 안전합니다. 증상 판단, 119/112 대체, 보험 정보, 개인 진료기록 조회를 요구하지 않습니다.

## 예상 흐름

UMMAYA는 안전 질문을 짧고 보이는 순서로 처리해야 합니다. 캠퍼스, 행정구역, 주소, 근처 표현이 있으면 먼저 장소를 해석하고, 그 다음 안전, 날씨, 도로, 응급, 병원 adapter 중 관련된 공개 lookup만 선택합니다.

```text
사용자가 장소와 안전 요구를 질문
  -> `locate`가 장소를 해석
  -> `find`가 공개 안전/의료 정보를 조회
  -> 답변이 출처, 결과, 최신성, 긴급 공식 경계를 설명
```

adapter가 없거나 공개 source가 요청을 뒷받침하지 못하면 자신 있는 추측을 하면 안 됩니다. UMMAYA는 누락된 경로를 설명하고 공식 응급 또는 공공서비스 채널로 넘겨야 합니다.

## 좋은 답변의 구성

좋은 답변은 공개 evidence와 긴급 조언을 분리합니다. 어떤 공개 source나 adapter가 결과를 만들었는지, 그 결과가 무엇을 뒷받침하는지, 어떤 불확실성이 남는지, 긴급 상황이면 사용자가 무엇을 해야 하는지 말해야 합니다.

예를 들어 가까운 병원 목록을 찾았더라도 실시간 수용 가능성, 구급 출동, 의료 triage는 공식 응급 채널에서 처리해야 한다고 말해야 합니다. 이 구분이 공개 lookup을 임상 판단으로 오해하지 않게 만듭니다.

## UMMAYA가 하면 안 되는 것

UMMAYA는 tool result가 증명하지 않은 의료, 응급, 개인기록 주장을 하면 안 됩니다. live source가 해당 상태를 제공하지 않는 한 병원이 사용자를 받아준다고 말하면 안 됩니다. 즉각적인 위험이 암시되는 상황에서 공식 응급 연락을 지연시키는 안내도 하면 안 됩니다.

안전한 표현은 구체적입니다. `공개 정보 기준`, `source가 반환한 결과`, `가능 여부는 변동될 수 있음`, `긴급하면 119 또는 공식 채널 이용`처럼 말해야 합니다. 위험한 표현은 근거 없이 권위적입니다. `안전합니다`, `이 병원이 받아줍니다`, `응급 서비스가 필요 없습니다` 같은 문장은 evidence가 없으면 금지됩니다.

## 복구

흐름이 멈춰도 사용자는 다음 행동을 얻어야 합니다. UMMAYA는 어떤 evidence가 없었는지, stop이 no adapter인지, no live result인지, protected data인지, official Handoff인지 설명하고 이어갈 공식 경로를 알려야 합니다.

안전 영역에서는 정직한 stop이 제품의 일부입니다. 고위험 상황에서 거짓 확실성을 만드는 것보다 “UMMAYA는 공개 안내를 찾았지만 응급 수용 가능성은 확인할 수 없습니다”라고 말하는 편이 더 안전합니다.
